# CLAUDE CODE PROMPT: PROJECT START

**Role:** Expert Game Developer & Software Architect.
**Project Name:** "Resilience Light" (Світло Стійкості).
**Genre:** Economic strategy, crisis management simulator, state survival.
**Platform:** Web Browser (HTML5 Canvas + LocalStorage, No Backend).
**Visual Style:** Military-tech, "Digital Alert Map", Dark Mode, Glassmorphism, neon accents.

## 1. TECH STACK & ARCHITECTURE
* **Map/Graphics Engine:** Pure HTML5 Canvas API or PixiJS (WebGL). **Strict Rule: No heavy DOM frameworks (Vue/React) for rendering the game map and active objects.** UI overlays can be basic HTML/CSS over the canvas.
* **Interaction:** "Hit Areas" (polygons) overlaying the Canvas to handle clicks on specific regions.
* **Environment:** Pure Client-Side Web Application. Game state and saves are managed entirely via browser `localStorage` (or `IndexedDB` for larger saves), with no backend server.
* **Data:** GeoData (GeoJSON) for the 26 region borders, JSON for infrastructure and cities. These are static assets fetched via standard `fetch()` API. Mercator projection required (converting `lat/lon` into `x/y` coordinates on the Canvas).

## 2. CORE GAME LOOP
The game is divided into two strict phases:

### Phase 1: DAY (Strategy & Planning)
* **State:** Time is paused. No enemy attack animations occur.
* **Player Actions:** * Allocate budget for infrastructure repairs.
    * Relocate mobile Air Defense (AD) units.
    * Open "Points of Invincibility" (social shelters) and build protective structures (concrete covers/underground facilities).
    * Configure power grid routing (managing power flows between regions).
    * Implement rolling blackouts to preserve grid stability and prevent overload.
* **Controls:** Clicking on regions, opening information panels, adjusting priorities. State changes are saved to `localStorage`.

### Phase 2: NIGHT (Simulation & Consequences)
* **State:** Management UI is locked. The in-game night timer runs.
* **Actions:**
    * Visualize enemy attacks (missiles and drones represented as vectors flying toward targets).
    * Automatic engagement by player's Air Defense (interceptions).
    * Calculate hits: If an object is destroyed, its state changes, and connected grid lines are severed.
    * Visualize blackouts (regions go dark on the map in real-time).
    * Simulate random events: Grid overloads, accidents, and fires.
* **Transition:** Morning generates a daily report (damages, casualties, shifts in International Support) and auto-saves the game.

## 3. BUSINESS LOGIC & MATH

### 3.1. Resources & Metrics
* **Money (₴):** Global and regional budgets. Generated by population taxes.
* **International Support (%):** Slowly decays over time. Spikes if there are civilian casualties (bombings of residential areas). High support unlocks advanced Western AD systems and financial grants.
* **Depression (0-100%):** The primary loss condition metric.

#### Depression Formula Parameters
**Daily Depression Delta (Δdepression per day):**
```
Δdepression = base_per_day 
            + (bombing_per_casualty × casualties_count)
            + (bombing_per_infra × infrastructure_hits_count)
            + (no_power_penalty × no_power_factor × winter_multiplier)
            + (no_heat_penalty × no_heat_factor × winter_multiplier)
            - (social_project_bonus × active_projects_count)
            - (decay_rate × current_depression / 100)

Final: depression = clamp(depression + Δdepression, 0, 100)
```

**Parameter Values:**
* `base_per_day` = **0.5%** — Natural baseline increase from ongoing crisis stress
* `bombing_per_casualty` = **2.0%** — Depression increase per civilian casualty (residential bombing)
* `bombing_per_infra` = **0.8%** — Depression increase per destroyed infrastructure object (power plants, substations)
* `no_power_penalty` = **3.0%** — Base penalty when region has no electricity
* `no_heat_penalty` = **5.0%** — Base penalty when region has no heating (applies only to CHP-dependent regions)
* `winter_multiplier` = **2.5** in winter months (Nov–Mar), **1.0** otherwise — Seasonal scaling for power/heat penalties
* `social_project_bonus` = **1.5%** per active project level — Depression reduction from social infrastructure investments
* `decay_rate` = **0.1** — Natural decay coefficient (depression slowly improves over time when conditions stabilize)
* **Clamping Rule:** Depression is always bounded to **[0%, 100%]**

**Factor Calculations:**
* `no_power_factor`:
  * **1.0** if region has **NO power** (hasPower = false)
  * **0.0** if region has power (hasPower = true)
* `no_heat_factor`:
  * **1.0** if region has **NO heat** AND has CHP dependency (hasHeat = false, CHP exists in region)
  * **0.0** otherwise (region has heat OR no CHP infrastructure present)

**Persistence/Decay Behavior:**
* Depression **increases** from negative events (bombings, blackouts) immediately on the day they occur
* Depression **decays naturally** at rate `decay_rate × current_depression / 100` per day (exponential decay)
  * Example: At 80% depression, natural decay = 0.1 × 80 / 100 = **0.08%/day**
  * Example: At 20% depression, natural decay = 0.1 × 20 / 100 = **0.02%/day**
* **Social projects provide sustained reduction** as long as they remain funded/active
* **Bombing effects persist** until decay naturally reduces them (no instant reset)

**Example Calculation (Winter Day with Power Outage):**
```
Scenario: Day 15, Winter (Jan), Region: Kyiv Oblast
- Current depression: 45%
- Events today: 3 casualties, 1 substation destroyed, no power, no social projects active
- Season: Winter (winter_multiplier = 2.5)

Δdepression = 0.5                          // base_per_day
            + (2.0 × 3)                    // bombing_per_casualty × casualties
            + (0.8 × 1)                    // bombing_per_infra × infrastructure_hits
            + (3.0 × 1.0 × 2.5)           // no_power_penalty × no_power_factor × winter_multiplier
            + (5.0 × 0.0 × 2.5)           // no_heat_penalty × no_heat_factor × winter_multiplier (no CHP dependency)
            - (1.5 × 0)                    // social_project_bonus × active_projects
            - (0.1 × 45 / 100)            // decay_rate × current_depression / 100

Δdepression = 0.5 + 6.0 + 0.8 + 7.5 + 0.0 - 0.0 - 0.045
            = 14.755%

New depression = clamp(45 + 14.755, 0, 100) = 59.755% → rounded to 59.8%
```

**Implementation Note:** Use named parameters exactly as specified (Base, bombing_per_casualty, bombing_per_infra, no_power_penalty, no_heat_penalty, winter_multiplier, social_project_bonus, decay_rate) in configuration files or constants for easy tuning during testing.

### 3.2. Population (Pool of 10,000 Units)
10,000 abstract units are distributed across the 26 regions. Divided into 3 classes:
* **Autonomous (10%):** Have personal generators/resources. If local depression > 60%, they migrate abroad (permanent loss of tax revenue).
* **Mobile (50%):** The core tax base. They migrate to neighboring "green" (safe/powered) regions when depression gets too high, causing grid strain in the receiving region.
* **Rooted (40%):** Highly vulnerable. Cannot migrate and depend entirely on state infrastructure and social projects.

### 3.3. Power Grid & Objects
Infrastructure is parsed from static `infrastructure.json` with exact coordinates:
* **Global Generation (Tier 1/2):** NPPs (Nuclear - baseload), HPPs/PSPPs (Hydro - peak balancing), TPPs (Thermal - balancing).
* **Local Generation (CHPs/Factories):** Have a `heat: true` property. If these stop in winter, the region's depression multiplier increases drastically due to freezing.
* **Substations (750kV / 330kV):** Nodes in the energy graph. If a substation is destroyed, the power line breaks, isolating regions from the main grid.

### 3.4. Enemy AI
Scripted AI with escalating phases:
1.  Mass chaotic strikes on cities (Initial Shock).
2.  Targeted strikes on main substations and generation facilities (Systemic Blackouts).
3.  Pinpoint drone swarms to exhaust AD interceptor stocks before launching ballistics.

## 4. DATA MODELS
The system expects the following static input files in an `/assets` folder:
1.  `ukraine_regions.geojson`: Vector boundaries of the regions (for custom projection).
2.  `cities.json`: `[ { id, name, lat, lon, pop, tier } ]` - Used to render glowing city lights.
3.  `infrastructure.json`: `[ { id, name, category, type, lat, lon, capacity, status, heat, voltage } ]`.
4.  `rivers.json`: River geometry for map background aesthetics.

### 4.1. Static Asset Schemas & Validation

#### Data File Version & Migration Policy
* Each static JSON file must include or reference a `_version` field (e.g., `"_version": "1.0.0"`)
* Version format: MAJOR.MINOR.PATCH (semver)
* **Migration Strategy:** If schema version > expected version:
  * Log warning and attempt coercion to expected schema
  * If schema version < expected version: Use migration adapters in `src/assets/migrations/` directory
  * If coercion fails: Fail-fast with clear error message listing required fields

#### TypeScript Interfaces (Canonical Source of Truth)
These are defined in [src/types/index.ts](src/types/index.ts) and govern runtime validation:

**ukraine_regions.geojson** (GeoJSON FeatureCollection)
```typescript
interface OblastFeature {
  type: 'Feature';
  properties: {
    id: string;        // Unique region identifier (e.g., "kyiv", "kharkiv")
    name: string;      // Ukrainian region name (e.g., "Київська", "Харківська")
  };
  geometry: GeoPolygon | GeoMultiPolygon;  // Polygon or MultiPolygon for region boundary
}

interface FeatureCollection<OblastFeature> {
  type: 'FeatureCollection';
  features: OblastFeature[];
}
```
**Semantic Constraints:**
* Exactly 26 features required (one per Ukrainian oblast + Kyiv city)
* Coordinates in WGS84 (EPSG:4326), format [lon, lat]
* All polygons must be closed (first and last point identical)
* No coordinate duplicates within a polygon ring

**cities.json** (Array of City objects)
```typescript
interface City {
  id: number;          // Unique city identifier (no duplicates)
  name: string;        // City name in Ukrainian (≥1 character, ≤100 characters)
  lat: number;         // Latitude (WGS84): -90 to +90
  lon: number;         // Longitude (WGS84): -180 to +180
  population: number;  // Population count in thousands (≥0, integer)
  tier: number;        // Strategic tier (enum: 1=megacity, 2=major, 3=secondary, 4=tertiary)
}
```
**Semantic Constraints:**
* `tier` must be integer in range [1, 4]
* `population` must be non-negative integer representing thousands
* All cities must be within Ukraine's geographic bounds (48°-52.5°N, 22°-40.5°E)
* No duplicate coordinates (within 1 km tolerance for rounding)

**infrastructure.json** (Array of InfrastructureObject)
```typescript
type InfrastructureType = 'nuclear' | 'hydro' | 'thermal' | 'substation' | 'chp';
type InfrastructureStatus = 'active' | 'damaged' | 'destroyed' | 'occupied';
type PowerLineVoltage = '750kV' | '330kV' | 'chp';

interface InfrastructureObject {
  id: string;             // Unique identifier (e.g., "npp_rivne", "sub750_kyiv")
  name: string;           // Ukrainian name (≥1, ≤150 characters)
  type: InfrastructureType;
  lat: number;            // Latitude (WGS84): -90 to +90
  lon: number;            // Longitude (WGS84): -180 to +180
  capacity: number;       // Numeric capacity with type-specific units:
                          //  - nuclear/hydro/thermal/chp: MW (megawatts) [0-5000]
                          //  - substation: reserved for future (always 0 for now)
  status: InfrastructureStatus;
  heat?: boolean;         // Only valid for type='chp'. If true: generates district heat in winter
                          // Effect: Prevents depression multiplier spike in connected region during winter
                          // Default: false if omitted
  voltage?: PowerLineVoltage; // Only valid for type='substation'. Specifies grid tier:
                          //  - '750kV': Ultra-high voltage backbone (carries 50-100% of regional load)
                          //  - '330kV': High voltage distribution (carries 20-50% of regional load)
                          //  - 'chp': Local (rarely used, legacy)
}
```
**Semantic Constraints:**
* `type='substation'` MUST have `voltage` defined; capacity must be 0
* `type='chp'` MAY have `heat=true`; other types: heat field ignored if present
* All infrastructure must be within Ukraine's geographic bounds
* `capacity` for generators must be > 0; substations always 0
* `status='occupied'` means facility controlled by adversary: generates 0% power, cannot be repaired

**rivers.json** (Array of River objects)
```typescript
interface River {
  name: string;        // River name (e.g., "Дніпро", "Дністер")
  path: [number, number][]; // Array of [lat, lon] points (WGS84) forming river geometry
                        // Minimum 2 points required for valid line
}
```
**Semantic Constraints:**
* Each point must be [lat, lon] tuple (NOT [lon, lat] like GeoJSON)
* All coordinates within Ukraine's geographic bounds
* River paths must be continuous (no orphaned segments)

#### Runtime Validation Strategy
**Approach:** TypeScript + JSON Schema (using `ajv` validator)
**Location:** [src/assets/dataValidation.ts](src/assets/dataValidation.ts)

**Validation Pipeline:**
1. **Fetch Phase:** Asset loader calls `validateAsset(filename, data)` immediately after fetch
2. **Schema Check:** ajv validates against canonical JSON Schema (generated from TypeScript interfaces)
3. **Semantic Check:** Custom validators check constraints (e.g., 26 regions, bounds, no duplicates)
4. **Coercion Phase:** If validation passes with warnings, attempt type coercion (e.g., string→number)
5. **Result:** Return `{ valid: boolean, data: T | null, errors: ValidationError[] }`

**Error Handling Policy:**
* **Fail-Fast (Default):** Any validation error halts game startup with user-facing error modal
* **Graceful Degradation (Optional Log-Only):** Non-critical assets (rivers, some cities) skip fail-fast
  * Log warnings to console, continue with subset of data
  * Example: Missing river.json → game renders without rivers, logs warning
* **Missing Files:** Fail-fast for critical files (regions, cities, infrastructure), graceful for rivers
* **Malformed JSON:** Fail-fast with clear syntax error

**Examples of Validation Steps:**

```typescript
// Example: Validate cities.json
const citiesData = await fetch('/assets/cities.json').then(r => r.json());
const validation = validateAsset('cities.json', citiesData);

if (!validation.valid) {
  // Show error modal with human-readable message
  showErrorModal({
    title: 'Data Load Error',
    message: `Failed to load cities.json:\n${validation.errors.map(e => e.message).join('\n')}`
  });
  throw new Error('Critical asset validation failed');
}

const cities: City[] = validation.data!;
```

```typescript
// Example: Validate infrastructure.json with custom semantic checks
validateInfrastructure(infraData): ValidationResult {
  // 1. Schema validation (ajv)

  //    - All substations have voltage defined
  //    - All capacities >= 0
  //    - All coordinates within bounds

```

**Default Values & Recovery:**
* `heat` field: Defaults to false if omitted
* `voltage` field: Required for substations; missing = validation error (no default)
* `status` field: Defaults to 'active' if omitted (assumes facility is operational)
* Invalid `tier`: Fallback to tier=4 (lowest) with warning log

#### Integration in Asset Load Pipeline
**Location:** [src/assets/AssetsManifest.ts](src/assets/AssetsManifest.ts)

The loader flow:
```
fetch(...json) 
  → JSON.parse() 
  → validateAsset(...) [CANONICAL VALIDATION STEP]
  → Type cast to TypeScript interface 
  → Return to MapRenderer
```

**Validation Step Reference Names (for logging/testing):**
* `validate_ukraine_regions` → Validates ukraine_regions.geojson (26 features, closed polygons, no overlaps)
* `validate_cities` → Validates cities.json (unique IDs, bounds, tier in [1-4], population ≥0)
* `validate_infrastructure` → Validates infrastructure.json (required fields, proper voltage for substations, capacity bounds)
* `validate_rivers` → Validates rivers.json (minimum 2 points per river, continuous paths, optional fail-soft)

**Testing Assertions** (developers should verify these when modifying schemas):
* Load with missing required field → Validation error with field name
* Load with invalid type (e.g., `tier: "1"` string instead of number) → Coerce or error per policy
* Load with out-of-bounds coordinate → Clear error message with coordinate value
* Load with valid but semantically inconsistent data (e.g., substation with non-zero capacity) → Warning log, correction applied or error
